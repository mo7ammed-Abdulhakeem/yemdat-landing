# v5.7.3 — Deleted members can re-register (email uniqueness fix) (2026-07-09)

Fixes a confusing dead-end reported on the live server: after an admin **deletes a
member**, that person could neither sign up again ("The email has already been
taken") nor sign in ("credentials do not match"), and no member showed in the admin
list. Root cause was **not** a soft delete — members are hard-deleted. Member
registration validated the email with `unique:users|unique:members`, so any email
that also belonged to a **staff/admin or trainer `User`** was rejected at signup, even
though the member row was truly gone. The message was misleading because it was
matching the `users` table, not `members`.

Member and admin/trainer accounts use **separate guards and tables by design**, so a
member is now allowed to register regardless of whether the email exists as a staff
user. After this change, "email already taken" at signup means only that a real
**member** already uses it.

## What's changed
- **Member signup no longer blocks emails that belong to staff/admin/trainer
  accounts.** A deleted member (or an admin) can register as a member again.
- Member login and the admin Members list were already correct — no change; this
  removes the cross-guard block that caused the "taken vs no-record" mismatch.

## Files
- **Edited:** `app/Http/Controllers/Auth/RegisterController.php` — email rule
  `unique:users|unique:members` → `unique:members` (dropped the cross-guard
  `unique:users` check).
- **Unchanged and intentional:** hard delete stays a hard delete; trainer promotion
  still "aborts when the email belongs to an admin"; `public/build/**` unchanged from
  v5.7.2 (no CSS/JS/Blade touched).

## Server deployment (Hostinger manual upload)
1. **Extract this bundle at the project root**, overwriting
   `app/Http/Controllers/Auth/RegisterController.php`.
2. **No migrations, no seeders, no `composer install`, no new dependencies.**
3. Run **`php artisan optimize:clear`**.
4. **Vite build unchanged** — no need to re-upload `public/build`.
5. Verify: with an admin/staff account on email X, go to `/register` and sign up a
   member using email X → it now proceeds to the OTP screen (no "already taken"); after
   verifying the OTP the member can sign in. A still-active member's email is still
   rejected as a duplicate.
