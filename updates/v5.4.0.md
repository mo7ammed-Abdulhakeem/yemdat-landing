# v5.4.0 тАФ Admin replies, certificate emailing & admin notifications (2026-06-07)

Closes the loop on inbound messages and certificates. Admins can now **reply to
contact messages and trainer applications from the panel** (with a tracked
status + reply history), **email certificates to members** (auto-sent on issue),
and get a **bell notification** when a new contact/trainer submission arrives.
Also adds prev/next navigation on record pages and a read-only Email Template
viewer.

## Admin replies (Filament)
- A **"Reply"** action on the Contacts and Trainer Requests tables **and** their
  view pages opens a compose modal (from name/email, subject, rich-text body),
  sends a branded email, records it in a **reply history**, and flips the record
  to **Replied**.
- **Status workflow** `new тЖТ replied тЖТ closed` on both record types: status
  badge column, status filter, "Mark as closed" / "Reopen" actions (single +
  bulk). Sending a reply never silently reopens a manually-closed record.
- Single shared code path `App\Actions\SendAdminReply` (send тЖТ record тЖТ set
  status; if delivery throws, nothing is recorded and status is unchanged).
- Default reply "from" name/email are editable at **Settings тЖТ Admin Replies**
  (`reply_from_name` / `reply_from_email`; fall back to `config('mail.from.*')`).
  The email is sent from the **authorized mailbox** with the chosen name; the
  typed address is set as **Reply-To** (keeps SPF/DKIM aligned on shared hosting).

## Certificate emailing
- **"Email to member"** / **"Resend email"** action on the certificate list +
  view (admin) and on the trainer panel's certificates, with a language picker;
  the rendered **PDF is attached**. Stamps `certificates.emailed_at`.
- **Auto-send on first issue:** issuing a certificate emails the member once
  (only on fresh creation, only when the `CertificateIssuedEmail` template is
  active). Mail failure is logged and **never blocks issuance**.
- **Bulk issue does NOT auto-email** (see note below) тАФ it issues silently to
  avoid sending dozens of synchronous emails in one request on shared hosting.
  Use the per-certificate "Email to member" action to deliver those.
- Single shared code path `App\Actions\Certificates\SendCertificateEmail`.

## Admin notifications (the bell)
- New contact message тЖТ notifies admins with the `messages` permission; new
  trainer application тЖТ notifies all admins. Rendered via Filament database
  notifications (`->databaseNotifications()` on the admin panel). Sending is
  wrapped so it can **never break the public form submission**.

## Record navigation & misc
- **Previous / Next** header actions on record pages (Contacts, Trainer
  Requests, Members view/edit, Certificates) via `App\Filament\Concerns\HasRecordNavigation`
  (admin panel only тАФ follows primary-key order, ignores active filters).
- Read-only **Email Template** view page (infolist) in the admin panel.

## Data model (additive migrations тАФ safe on existing rows)
- `contacts`: + `status` (string, default `new`, indexed).
- `trainer_requests`: + `status` (string, default `new`, indexed).
- `certificates`: + `emailed_at` (nullable timestamp).
- New `email_replies` (polymorphic `replyable`, from_name/email, subject, body,
  `sent_by` тЖТ users nullOnDelete).
- New `notifications` (Laravel's standard table тАФ required for the admin bell).

All five migrations are individually guarded (`hasTable` / `hasColumn`
early-return), so they are safe to run against the existing production DB and
safe to re-run.

## Server deployment (Hostinger manual upload)
1. Upload changed PHP files (Actions, Models, Mail, Support, Filament,
   Controllers) + the 5 new migrations.
2. `php artisan migrate --force` (adds `contacts.status`,
   `trainer_requests.status`, `certificates.emailed_at`; creates `email_replies`
   + `notifications`). **Required** тАФ without it the reply/status/bell features
   error or no-op in the panel.
3. `php artisan db:seed --class=EmailTemplateSeeder --force` (idempotent
   `firstOrCreate` by `mailable_class` тАФ adds the `CertificateIssuedEmail`
   template; existing templates untouched).
4. `php artisan optimize:clear` (re-`optimize` if you cache in prod тАФ all code is
   class-based, so `route:cache`/`config:cache` stay safe).
5. No `npm run build`, no `filament:assets:publish`.

**Operational note:** mail is sent synchronously (`QUEUE_CONNECTION=sync`) and
the SMTP relay is capped at ~100 emails/day. Auto-send-on-issue is one email per
certificate; bulk issue deliberately does **not** email. No new dependencies.
