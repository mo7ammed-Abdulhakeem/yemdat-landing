# v5.6.0 — Analytics dashboard rebuilt: filterable, sectioned, bilingual, mobile-friendly (2026-06-20)

The admin **Analytics** page (`/admin/analytics`) went from a flat grid of 7
member-demographic charts to an analyst-grade dashboard: a **global date-range
filter**, a KPI row with trend sparklines, **six collapsible sections**, and
**11 new insight widgets** that surface data the app already collected but never
showed (event attendance/completion, broadcast engagement, support pipeline,
certificate fulfilment). Fully **bilingual EN/AR + RTL** and **mobile-tuned**.
Backend-only — no migrations, no asset rebuild. Full suite green (173).

## What's new on the Analytics page
- **Global date-range filter** (last 7 / 30 / 90 / 365 days, all time, or a custom
  from/until). Persists across reloads; every range-aware widget reacts to it.
- **KPI row** (8 stat cards with sparklines): members + growth vs the previous
  equal period, new members, event registrations, active/upcoming events,
  certificates issued (flags any not yet emailed), email-verification rate, and
  new contact / trainer requests awaiting a first reply.
- **Six sections**, each collapsible: **Growth & acquisition** (signups trend +
  reach funnel) · **Event performance** (registrations over time, show rate,
  completion rate, demand by format/level, per-event outcomes, certificates
  issued-vs-emailed, top events, upcoming-events table) · **Member demographics**
  (gender, tier, education, countries, specialty — collapsed by default) ·
  **Engagement & communications** (broadcast open/unsubscribe stats + a
  per-campaign engagement table) · **Operations pipeline** (contacts & trainer
  requests by status).
- The **home Dashboard** stays light: KPI overview + member-growth pulse +
  upcoming-events table.

## Insights this exposes (for decision-making)
- **Show rate & completion rate** computed from `event_member.attended_at /
  completed_at` — content quality and where reminders/follow-ups are needed.
- **Member reach funnel** (total → email-verified → still-subscribed) — surfaces
  deliverability/activation problems before the next campaign.
- **Broadcast open rate & unsubscribes** per campaign — email-program health.
- **Certificates issued vs emailed** — a visible gap is a delivery backlog.
- **Inbox pipeline** (new/replied/closed) — support backlog at a glance.

## Performance & correctness
- **Killed a hidden 5-second poll.** Filament's `CanPoll` defaults every widget to
  `wire:poll.5s`, so all widgets were re-querying the DB every 5 seconds. Disabled
  centrally; widget queries are now cached 300s (keyed by active filters + locale).
- **Bilingual fixes:** `SetLocale` now syncs Carbon's locale so month names
  translate; week-bucket charts pin to Monday (Carbon's `startOfWeek()` is Saturday
  in Arabic, which previously zeroed out AR charts); broadcast engagement table uses
  `withCount` aggregates instead of the per-row accessors (avoids N+1).
- Member-reply **status labels are now translated** (Contacts / Trainer Requests
  badges localize too).

## Architecture
- Shared trait **`app/Filament/Widgets/Concerns/InteractsWithAnalytics.php`**:
  permission gate (`analytics`), no-poll, date-range resolution, granularity +
  zero-filled time-series + sparkline helpers, short-lived caching, and RTL-aware
  chart options. Reused by all 18 widgets.
- The Analytics page composes sections via an overridden `content()` using
  Filament `Section` + `Grid` schema components (`HasFiltersForm` for the filter).

## Server deployment (Hostinger manual upload)
1. Extract `updates/v5.6.0.zip` at the **project root** — overlays the new/changed
   PHP and `lang/` files (no migrations, no seeders).
2. `php artisan optimize:clear` (clears config/route/view/cache so the new widgets,
   translations, and middleware change take effect). Re-`optimize` afterwards if you
   cache in prod — all code is class-based, so route/config cache stay safe.
3. **No `npm run build`, no `public/build` upload** — no frontend assets changed;
   the charts use Filament's already-published widget assets.
4. No new dependencies (`composer install` not required).
