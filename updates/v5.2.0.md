# v5.2.0 — Trainer role + dedicated trainer panel (2026-06-05)

Admins can now **promote a community member into a trainer**. A trainer keeps their
normal Member account (public site, certificates, profile) and additionally logs into
a separate, locked-down **`/trainer`** dashboard where they manage only the events
they're assigned to teach.

## What a trainer can do
- Sign in to **`/trainer`** (the existing `/admin` is untouched and off-limits to
  trainers; admins can't reach `/trainer` either).
- **My Events** — see only events they're the assigned trainer of, **read-only**
  (cannot edit event details).
- **Registrations** — view attendees, mark attended/completed.
- **Certificates** — issue, email, download, and **revoke/reinstate** certificates for
  their own events (certificates record the trainer as the issuer).

## Architecture (why it's safe)
- A trainer is a **staff `User` with `role = 'trainer'`** linked to their `Member`
  (`members.user_id`). Reuses the web guard, so `IssueCertificate`'s `issued_by`
  (a User FK) and `FilamentUser` all keep working.
- **`User::canAccessPanel()` branches by panel id**: `trainer` panel ⇒ role
  `trainer`; everything else ⇒ `admin`/`super_admin`. Panel-level isolation means the
  11 existing `/admin` resources were **not touched** — no per-resource gating retrofit.
- Trainer resources also scope `getEloquentQuery()` to the acting trainer (defense in
  depth — another trainer's event/cert resolves to 404).

## Promotion & onboarding
- **"Make trainer"** action on the Member view page + members table: creates/links a
  `trainer` User (aborts if the email already belongs to an admin) and **emails a
  set-password link**. **"Revoke trainer"** unlinks + deletes the trainer user (events
  un-assign via the FK).
- The invite uses a **temporary *signed* route** to the trainer panel's Filament
  password-reset page (a plain `route()` URL is rejected 403). Trainers can also always
  self-serve via `/trainer/password-reset/request`.
- New `TrainerInviteEmail` mailable (`DynamicEmailTrait`) + a bilingual
  `TrainerInviteEmail` template in `EmailTemplateSeeder`.

## Event assignment
- Events gain a nullable **`trainer_id`** (→ users). The `/admin` event form has an
  "Assigned trainer" Select (lists trainer users); the events table shows a Trainer
  column. The free-text `lecturer_*` fields still drive public display.

## Data model (additive migrations)
- `members.user_id` (nullable FK → users, nullOnDelete) — activates the dormant
  `User::member()` link.
- `events.trainer_id` (nullable FK → users, nullOnDelete).
- `role` is a string column — `trainer` is just a new value (UserForm gains the option).

## Models
- `Member`: `user()` belongsTo, `isTrainer()`, `user_id` fillable.
- `User`: panel-aware `canAccessPanel()`, `trainedEvents()`, `isTrainer()`.
- `Event`: `trainer()` belongsTo, `trainer_id` fillable.

## Server deployment (Hostinger manual upload)
1. Upload changed PHP + the new `TrainerPanelProvider` + Trainer resources + Mail +
   2 migrations; ensure the provider line is in `bootstrap/providers.php`.
2. `php artisan migrate --force` (adds `members.user_id`, `events.trainer_id`).
3. `php artisan db:seed --class=EmailTemplateSeeder --force` (adds the invite template;
   idempotent — `firstOrCreate` by `mailable_class`).
4. `php artisan optimize:clear` (re-`optimize` if cached — the new panel's routes are
   class-based, `route:cache` stays safe). Filament assets are shared across panels —
   **no** `filament:assets:publish` needed. No `npm run build` (no public Blade/CSS).

No new dependencies. The invite link / token honors `auth.passwords.users.expire`
(default 60 min); trainers can request a fresh link from the trainer login page anytime.
