# v4.9.2 — Performance & production caching (Phase 5c) (2026-06-01)

## What changed

Phase 5c makes the app safe to run with Laravel's production optimizations and trims page weight.

- **`route:cache` now works** — the four route *action closures* (home `/`, `/lang/{locale}`,
  `/training`, and the 404 fallback) were the only thing blocking `php artisan route:cache`. They
  were moved to controllers (`HomeController`, `LanguageController`, `FallbackController`) and
  `/training` became a plain `Route::redirect`. (Route *group* closures are fine and untouched.)
- **`config:cache` is now safe** — two `env()` calls in application code (which return `null`
  once config is cached) were moved behind config:
  - `BROADCAST_DAILY_LIMIT` → `config('yemdat.broadcast_daily_limit')` (new `config/yemdat.php`).
  - `env('APP_NAME')` in `ContactUsAdminAlert` → `config('app.name')`.
- **Eager-loading** — the home page's latest-news query now eager-loads `author` (news index /
  post show already did), avoiding per-card author queries.
- **Image optimization** — event, post, and profile event-card images get `loading="lazy"` +
  `decoding="async"` (native, zero-dependency — safe on shared hosting where binary optimizers
  aren't available).

Net effect: production can now run `php artisan optimize` (route + config + view cache) without
breaking broadcasts or env-dependent code.

## Server deployment
1. Upload the files above + `config/yemdat.php`.
2. Keep `BROADCAST_DAILY_LIMIT` in `.env` (still read, now via config).
3. **Enable production caches:** `php artisan optimize` (or `route:cache` + `config:cache` +
   `view:cache`). Re-run after any future `.env` / route / config change.
   - To undo: `php artisan optimize:clear`.

No migrations, no new dependencies.
