# v5.5.0 — Broadcast targeting & stats, certificate UX + permissions, site animations (2026-06-10)

Three independent improvements: email **broadcasts can now target a membership
segment and show post-send stats**; **certificate issuing moved off the event
Edit form onto a dedicated View page** and the **Events/Certificates admin is now
gated on the `events` permission** (it previously wasn't); and the **public site
gained tasteful scroll-reveal animations**. All additive — full test suite green
(173).

## Broadcasts — audience targeting + post-send stats
- New audience types on a broadcast: **By Membership Tier** (Intern / Expert /
  Corporate — covers "experts only") and **Trainers Only** (members promoted to a
  trainer account). Unsubscribed members are excluded on every segment; event
  broadcasts now also honour the global unsubscribe.
- New **View page** per broadcast with a **Performance** panel (Recipients,
  Opened, Open rate, Unsubscribed) — surfacing open/unsubscribe data that was
  already collected but never shown. List gained toggleable Opened / Open rate /
  Unsub columns, a readable **Audience** label, and a View action.
- **Security:** added the missing `canView()` gate so a non-`broadcasts` admin can
  no longer read a broadcast via its view URL.
- Recipient query hardened: a blank tier targets nobody (not untiered members).

## Certificates — View-page UX + permission gating
- Admin certificate/attendance management **moved from the event Edit form to a
  new event View page** (mirrors the trainer panel). The events list gains an
  **"Attendees"** action; Edit is now event-details only.
- **Release-blocker fix:** `EventResource` and `CertificateResource` were
  ungated — *any* admin could manage events and issue/revoke certificates
  regardless of permission. Both now gate on the existing **`events`** permission
  (super-admins bypass). The `events` checkbox in the user editor is now enforced.
- Attendees table: removed an N+1 (event certificates loaded once); bulk issuance
  now logs *why* rows were skipped.

## Public site — scroll-reveal animations
- Subtle fade + rise as sections enter the viewport, with staggered reveals for
  card grids, across all public pages (home, about, vision/objectives, events,
  news, paths, membership, contact, be-a-trainer). **Admin/trainer panels
  untouched.**
- Progressive enhancement via a tiny `IntersectionObserver` (no new dependency):
  hiding is armed only when JS runs, so **JS off/blocked ⇒ content stays fully
  visible**; respects `prefers-reduced-motion`; vertical-only motion is RTL-safe.

## Data model (additive migration — safe on existing rows)
- `email_broadcasts`: `audience_type` widened `enum → string`; **+ `audience_value`**
  (nullable, holds the tier slug). Guarded `down()` normalises new values before
  reverting. Migration: `2026_06_10_000001_add_audience_targeting_to_email_broadcasts.php`.

## Server deployment (Hostinger manual upload)
1. Extract the bundle (`updates/v5.5.0.zip`) at the **project root** — overlays the
   changed PHP/Blade files and the new migration.
2. **Upload the Vite build** (`public/build/`) to **BOTH** public folders —
   `public_html/build` **and** `/yemdatlive/public/build` — or the public site
   renders unstyled (the animations CSS/JS live here).
3. `php artisan migrate --force` — adds `email_broadcasts.audience_value` and
   widens `audience_type`. **Required** for broadcast targeting; safe + guarded.
4. `php artisan optimize:clear` (re-`optimize` afterwards if you cache in prod —
   all code is class-based, so route/config cache stay safe).
5. No seeder needed. No new dependencies (`npm install` not required — assets are
   prebuilt in the bundle).

**Note:** after deploy, the `events` permission is now enforced — confirm any
admin who manages events/certificates has the **Events** permission ticked (super
admins are unaffected).
