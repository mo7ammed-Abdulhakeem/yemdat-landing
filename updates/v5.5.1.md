# v5.5.1 — Fix: admin media uploads go to the public disk + proper folders (2026-06-10)

Hotfix for broken cover images. Admin **`FileUpload` fields were saving to the
private `local` disk** (Filament's resolved default here), so newly uploaded
event covers, lecturer photos, post/news images, learning-path covers and email
banners were written to `storage/app/<file>` — never web-served — and 404'd on
the public site (`asset('storage/…')` reads the **public** disk). Old images kept
showing only because they were legacy files already in `storage/app/public/…`.

## Fix
Pinned every media upload field to `->disk('public')` and its folder, matching the
existing storage layout (the certificate-template field already did this):

- `EventForm`: cover → `public/events`, lecturer photo → `public/lecturers`
- `PostForm`: image → `public/posts`
- `LearningPathForm`: image → `public/paths`
- `EmailTemplateForm`: banner → `public/email-banners`

New uploads now land in the same folders as before and display correctly,
independent of any `.env` setting.

## Files
- Edited: `app/Filament/Resources/Events/Schemas/EventForm.php`,
  `app/Filament/Resources/Posts/Schemas/PostForm.php`,
  `app/Filament/Resources/LearningPaths/Schemas/LearningPathForm.php`,
  `app/Filament/Resources/EmailTemplates/Schemas/EmailTemplateForm.php`.

## Server deployment (Hostinger manual upload)
1. Extract `updates/v5.5.1.zip` at the **project root** (overlays the 4 form files).
2. `php artisan optimize:clear` (or at least `config:clear`, `view:clear`).
3. **No** `npm run build`, **no** asset re-upload, **no** migration.

Optional (already applied): `FILAMENT_FILESYSTEM_DISK=public` in `.env` fixes the
disk globally too; with this code it's redundant but harmless.
